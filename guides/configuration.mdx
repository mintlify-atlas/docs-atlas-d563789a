title: "Configuring Blocklet Server: Ports, SSL, and Storage"
description: "Configure Blocklet Server for production: set ports, SSL, storage backends, and resource limits via config.yml and ABT_NODE_* environment variables."
---

Blocklet Server reads its configuration from a `config.yml` file generated by `blocklet server init`, supplemented by environment variables. You set these values once during initialization and adjust them over time as your deployment requirements evolve. Understanding the priority order and the categories of settings available gives you full control over how the server behaves in production.

## Configuration priority

When the same setting can be expressed in multiple places, Blocklet Server resolves conflicts using this order — highest to lowest:

1. **Environment variables** (`ABT_NODE_*`) — always win
2. **`config.yml`** — the server's primary configuration file
3. **Built-in defaults** — used when neither of the above sets a value

Set sensitive or environment-specific values (secrets, database URLs) as environment variables. Use `config.yml` for stable structural settings like ports and routing.

## The config.yml file

`blocklet server init` creates a `.blocklet-server/config.yml` file in the directory where you run it. A minimal production configuration looks like this:

```yaml config.yml
node:
  name: My Blocklet Server
  description: Production server
  version: 1.16.15
  port: 8089
  routing:
    provider: nginx
    https: true
    httpPort: 80
    httpsPort: 443
    adminPath: /.well-known/service/admin
    maxUploadFileSize: 100
  runtimeConfig:
    daemonMaxMemoryLimit: 2048
    blockletMaxMemoryLimit: 512
  database:
    engine: sqlite

blocklet:
  port: 8089

env:
  ABT_NODE_LOG_LEVEL: info
```

The `env:` section lets you set any `ABT_NODE_*` variable that Blocklet Server picks up on startup — useful for Docker deployments where you want all configuration in one file.

## Network and ports

<Note>
Ports 80 and 443 are the defaults for HTTP and HTTPS respectively. If these are in use, `blocklet server init` will automatically select available alternatives.
</Note>

| Variable | Default | Purpose |
|---|---|---|
| `ABT_NODE_HTTP_PORT` | `80` | Public HTTP port for the nginx gateway |
| `ABT_NODE_HTTPS_PORT` | `443` | Public HTTPS port for the nginx gateway |
| `ABT_NODE_PORT` | `8089` | Internal port the server daemon listens on |
| `ABT_NODE_BLOCKLET_PORT` | `8090` | Starting port assigned to blocklet processes |
| `ABT_NODE_HOST` | `0.0.0.0` | Host address the server binds to |

To run on non-standard ports, pass these at init time or set them as environment variables before starting:

```bash
export ABT_NODE_HTTP_PORT=8080
export ABT_NODE_HTTPS_PORT=8443
blocklet server start
```

## Authentication and access control

<Warning>
The variables marked "system-generated" (`ABT_NODE_SK`, `ABT_NODE_SESSION_SECRET`, `ABT_NODE_TOKEN_SECRET`) are created automatically by `blocklet server init`. Do not set them manually unless you are restoring a server from backup.
</Warning>

| Variable | Default | Purpose |
|---|---|---|
| `ABT_NODE_IP_WHITELIST` | None (all IPs) | Comma-separated IPs or CIDR ranges allowed to access the admin dashboard |
| `ABT_NODE_DOMAIN_WHITELIST` | None | Comma-separated domains permitted for CORS |
| `ABT_NODE_DOMAIN_BLACKLIST` | None | Comma-separated domains to block at the gateway |
| `ABT_NODE_SESSION_TTL` | `3600` | Session lifetime in seconds |
| `ABT_NODE_SESSION_CACHE_TTL` | `300` | Session cache lifetime in seconds |
| `ABT_NODE_ADMIN_PATH` | `/.well-known/service/admin` | URL path prefix for the admin dashboard |
| `ABT_NODE_ALLOW_BLOCKLET_IFRAME_EMBEDDING` | `0` | Set to `1` to allow blocklets to be embedded in iframes on other domains |
| `ABT_NODE_TRUSTED_SOURCES` | None | Comma-separated store URLs trusted for blocklet installation in serverless mode |

To lock the admin dashboard to your office IP range:

```bash
export ABT_NODE_IP_WHITELIST="203.0.113.0/24,198.51.100.5"
```

## Database and storage

By default Blocklet Server uses an embedded SQLite database. For production deployments with higher concurrency you can switch to PostgreSQL.

<Tabs>
  <Tab title="SQLite (default)">
    ```yaml config.yml
    node:
      database:
        engine: sqlite
    ```

    The SQLite database file lives at `{DATA_DIR}/core/db-cache.db`. No additional setup is required.
  </Tab>
  <Tab title="PostgreSQL">
    ```bash
    export ABT_NODE_POSTGRES_URL="postgresql://user:password@localhost:5432/blocklet_server"
    ```

    When `ABT_NODE_POSTGRES_URL` is set, Blocklet Server automatically uses PostgreSQL instead of SQLite.
  </Tab>
  <Tab title="Redis cache">
    ```bash
    export ABT_NODE_CACHE_REDIS_URL="redis://localhost:6379"
    ```

    Redis is used as a distributed cache layer. When not set, an in-memory cache is used instead.
  </Tab>
</Tabs>

| Variable | Default | Purpose |
|---|---|---|
| `ABT_NODE_POSTGRES_URL` | None | PostgreSQL connection URL; SQLite is used when absent |
| `ABT_NODE_CACHE_REDIS_URL` | None | Redis connection URL for distributed caching |
| `ABT_NODE_CACHE_SQLITE_PATH` | `{DATA_DIR}/core/db-cache.db` | Path to the SQLite cache file |
| `ABT_NODE_NO_CACHE` | `0` | Set to `1` to disable caching entirely |

## Logging

| Variable | Default | Purpose |
|---|---|---|
| `ABT_NODE_LOG_DIR` | `{DATA_DIR}/logs/_abtnode` | Directory for server log files |
| `ABT_NODE_LOG_LEVEL` | `info` | Log verbosity: `error`, `warn`, `info`, or `debug` |

Set `ABT_NODE_LOG_LEVEL=debug` temporarily when troubleshooting; revert to `info` in normal operation to avoid filling disk.

## Docker resource limits

When blocklets run as Docker containers inside Blocklet Server, the following variables cap resource usage per container:

| Variable | Default | Purpose |
|---|---|---|
| `ABT_NODE_DOCKER_MEMORY` | `512m` | Memory limit per Docker container |
| `ABT_NODE_DOCKER_CPUS` | `1` | CPU limit per Docker container |
| `ABT_NODE_DOCKER_DISK_SIZE` | `10g` | Disk size limit per Docker container |
| `ABT_NODE_NOT_ALLOW_DOCKER` | `0` | Set to `1` to disable Docker support entirely |

## Performance limits

| Variable | Default | Purpose |
|---|---|---|
| `ABT_NODE_MAX_CLUSTER_SIZE` | `4` | Maximum number of cluster worker instances |
| `ABT_NODE_AUTO_BACKUP_INTERVAL` | `28800` (8 hours) | Seconds between automatic blocklet backups |
| `ABT_NODE_ENSURE_RUNNING_HIGH_LOAD_CPU` | `80` | CPU % threshold that triggers high-load mode |
| `ABT_NODE_ENSURE_RUNNING_HIGH_LOAD_MEMORY` | `80` | Memory % threshold that triggers high-load mode |
| `ABT_NODE_ENSURE_RUNNING_HIGH_LOAD_DISK` | `90` | Disk % threshold that triggers high-load mode |
| `ABT_NODE_RUNTIME_MONITOR_DISABLED` | unset | Set to `1` to disable the runtime health monitor |

## Applying configuration changes

Most environment variable changes take effect only after restarting the server:

```bash
blocklet server stop
blocklet server start
```

To apply changes from `config.yml` to the running database state, use the `--update-db` flag on the next start:

```bash
blocklet server start --update-db
```
