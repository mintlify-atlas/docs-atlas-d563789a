Blocklet Server uses decentralized identity (DID) as its authentication foundation. Instead of usernames and passwords, users authenticate with a DID Wallet—a mobile or browser-based app that holds their cryptographic identity. This approach is passwordless, phishing-resistant, and works consistently across every blocklet installed on a server. As a blocklet developer, you get a set of SDK tools that connect your app into this system without having to implement the underlying DID protocol yourself.

## What is a DID?

A DID (Decentralized Identifier) is a globally unique identifier anchored to a cryptographic key pair. The holder of the private key can prove ownership of the DID by signing a challenge—no central authority is involved. In Blocklet Server:

- **The server** has a DID that identifies the node (`Server.did`)
- **Each blocklet** has two DID-related identifiers: an `appPid` (the primary DID from `blocklet.yml`) and an `appDid` (a wallet address used for signing transactions)
- **Each user** is identified by their wallet's DID

When a user signs into a blocklet, they scan a QR code with their DID Wallet, sign a challenge, and the server issues a session token. The user's DID becomes their persistent identity across all blocklets on that server.

## How users authenticate

The standard authentication flow uses DID Connect, a protocol for wallet-based sign-in:

<Steps>
  <Step title="Display the QR code">
    Your blocklet's frontend renders a QR code generated by `WalletAuthenticator`. The code contains a signed challenge that the wallet can verify.
  </Step>
  <Step title="User scans with their DID Wallet">
    The user opens their DID Wallet app, scans the QR code, reviews the app information, and approves the sign-in. The wallet signs the challenge with the user's private key and sends the response back to your server.
  </Step>
  <Step title="Server creates a session">
    `WalletHandler` receives the wallet's response, verifies the signature, and creates a session. A `login_token` cookie is set in the browser.
  </Step>
  <Step title="Session verified on subsequent requests">
    The `session` middleware reads the `login_token` cookie on each request, validates it, and attaches a `SessionUser` object to `req.user`. Route handlers can then read `req.user` to get the authenticated user's profile.
  </Step>
</Steps>

## Setting up DID Connect in a blocklet

Use `WalletAuthenticator` and `WalletHandlers` from `@blocklet/sdk`:

```javascript
import path from 'path';
import AuthStorage from '@arcblock/did-connect-storage-nedb';
import { WalletAuthenticator } from '@blocklet/sdk';
import { WalletHandlers } from '@blocklet/sdk';

// Initialize the authenticator (reads blocklet metadata automatically)
export const authenticator = new WalletAuthenticator();

// Initialize the handler with a storage adapter
export const handlers = new WalletHandlers({
  authenticator,
  tokenStorage: new AuthStorage({
    dbPath: path.join(process.env.BLOCKLET_DATA_DIR, 'auth.db'),
  }),
});

// Mount the handlers on your Express app
app.use('/api/did/auth', handlers);
```

`WalletAuthenticator` reads your blocklet's metadata automatically so the wallet displays the correct app name, description, and icon when the user scans the QR code. You can override this with a custom `appInfo` function:

```javascript
const authenticator = new WalletAuthenticator({
  async appInfo() {
    return {
      name: 'My Custom App Name',
      description: 'A custom description shown in the wallet.',
      icon: 'https://my-app.com/logo.png',
    };
  },
});
```

## Protecting routes with the session middleware

The `session` middleware from `@blocklet/sdk/middlewares/session` reads the `login_token` cookie and populates `req.user`:

```javascript
import session from '@blocklet/sdk/middlewares/session';

// Apply globally (req.user is undefined for unauthenticated requests)
app.use(session());

// Apply to a specific route in strict mode (returns 401 if not authenticated)
app.get('/api/profile', session({ strictMode: true }), (req, res) => {
  res.json(req.user);
});
```

The middleware supports multiple authentication strategies:

| Option | Default | Description |
| --- | --- | --- |
| `strictMode` | `false` | Return `401` if no valid credential found; otherwise call `next()` with no user |
| `loginToken` | `true` | Accept JWT login tokens from DID Connect (read from `login_token` cookie) |
| `accessKey` | `false` | Accept long-lived access keys for programmatic access (CI/CD, scripts) |
| `componentCall` | `false` | Accept signed requests from other components (`x-component-sig` header) |
| `signedToken` | `false` | Accept a temporary signed JWT passed as a query parameter |
| `signedTokenKey` | `"__jwt"` | Query parameter name for `signedToken` authentication |

## RBAC: roles and permissions

Blocklet Server's authorization model is role-based. Each user who signs in to a blocklet is assigned a role. The `auth` middleware from `@blocklet/sdk/middlewares/auth` checks roles and permissions before allowing access:

```javascript
import auth from '@blocklet/sdk/middlewares/auth';

// Only users with the 'admin' role can access this route
router.get('/dashboard', auth({ roles: ['admin'] }), (req, res) => {
  res.json({ message: 'Welcome, admin.' });
});

// Require subscriber role + email KYC + non-programmatic auth method
router.post(
  '/update-subscription',
  auth({
    roles: ['subscriber'],
    kyc: ['email'],
    methods: ['loginToken', 'signedToken'],
  }),
  (req, res) => {
    res.json({ success: true });
  }
);
```

The `auth` middleware checks conditions in this order. If any check fails, the request is rejected immediately:

1. **Authentication**: Is the user signed in? (401 if not)
2. **Roles**: Does the user have at least one of the required roles? (403 if not)
3. **Permissions**: Does the user's role have at least one of the required permissions? (403 if not)
4. **KYC**: Has the user completed required verification (email, phone)? (403 if not)
5. **Methods**: Was the session established with an allowed method? (403 if not)

## How blocklets inherit server authentication

Blocklet Server manages a single user store per server. When a user signs into any blocklet on a server, their identity and role are stored at the server level and shared across all blocklets. This means:

- A user does not need to sign in separately to each blocklet
- Role assignments made by the server admin apply server-wide
- Each blocklet can define its own permission scoping on top of the inherited roles

The `BlockletService` SDK client handles this transparently: all API calls from your blocklet to the server use the blocklet's own credentials, and the server enforces role checks based on the requesting user's session.

## The BlockletService SDK client

`BlockletService` is the primary interface for a blocklet's server-side code to interact with Blocklet Server's user management, session, and team APIs. It wraps GraphQL queries and HTTP requests into a promise-based API and authenticates automatically using the blocklet's credentials.

```javascript
import { BlockletService } from '@blocklet/sdk';

const client = new BlockletService();

// Get the blocklet owner's profile
const { user } = await client.getOwner();

// Get a user by DID
const { user } = await client.getUser({ did: 'z1abc...' });

// List all users
const { users } = await client.getUsers({ pagination: { skip: 0, first: 20 } });
```

`BlockletService` automatically sets the `x-access-blocklet` and `x-component-did` headers so the server can identify which blocklet is making the call. You do not need to manage authentication headers manually.

## Session management

Sessions are JWT-based tokens stored in the `login_token` browser cookie. Two key settings control session lifetime:

```typescript
export type BlockletSettings = {
  session: {
    ttl: number;      // Session lifetime in seconds
    cacheTtl: number; // How long session data is cached before re-validation
  };
  // ...
};
```

These settings are accessible at runtime via `window.blocklet.settings.session` on the client side and via the `BlockletService` API on the server side.

The session middleware caches validated sessions for `cacheTtl` seconds to avoid re-validating the same token on every request. If `enableBlacklist` is enabled in blocklet settings, the middleware additionally checks that the token has not been revoked before accepting it.

## Authentication method reference

| Method | Use case | How credentials are passed |
| --- | --- | --- |
| `loginToken` | Standard user sign-in via DID Wallet | `login_token` cookie (JWT) |
| `accessKey` | Programmatic access (CI/CD, scripts) | `login_token` cookie (access key format) |
| `componentCall` | Secure inter-component calls | `x-component-sig` request header |
| `signedToken` | Temporary delegated access | `?__jwt=...` query parameter |

<Note>
All four methods are supported by the same `session` middleware. Enable the methods your blocklet needs by passing the corresponding options: `session({ accessKey: true, componentCall: true })`.
</Note>
